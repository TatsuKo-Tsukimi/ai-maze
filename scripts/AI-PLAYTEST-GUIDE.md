# ClawTrap AI 对战测试方案

## 概述

本方案使用 Claude sub-agent 作为玩家和 villain 进行真实语义级别的游戏测试。与概率模拟器的区别：

| | 概率模拟 (realistic-sim.js) | AI对战测试 |
|---|---|---|
| 试炼回答 | `Math.random() < 0.85` | agent写2-3句真实文字，villain做语义判定 |
| 反问内容 | `Math.random() < 0.40` | agent写反问文本，villain自评能否回答 |
| 移动决策 | 加权随机 | agent看小地图推理选方向 |
| 诱惑决策 | 基于profile概率 | agent根据当前HP/碎片/策略做判断 |
| 适用场景 | 数值平衡验证(100+局) | 体验质量验证(5-10局) |

两种测试互补，不互相替代。

## 架构

```
ai-play-engine.js          → 生成游戏场景JSON(迷宫+66回合状态)
        ↓
Claude sub-agent (×5并行)  → 读JSON，逐回合做决策，写试炼回答
        ↓                     同时扮演villain做语义判定
汇总agent                  → 收集10局结果，输出总结
```

## 步骤

### 1. 生成游戏场景

```bash
cd C:/Users/tatsuya/Documents/ClawTrap
for i in $(seq 1 10); do
  node scripts/ai-play-engine.js $i > /tmp/game${i}.json
done
```

每个JSON包含：
- 迷宫参数（尺寸、出生点、出口、BFS最短路径）
- 66回合的完整状态（位置、小地图、可选方向、卡牌、试炼题目、碎片/推墙可用性、突发事件可能性）
- 游戏规则摘要

### 2. 启动5个并行agent

每个agent玩2局，使用不同策略。推荐策略分配：

| Agent | 局 | 策略 | 测试重点 |
|-------|-----|------|---------|
| 1 | 1-2 | 自由风格 | 整体体验、试炼对话质量 |
| 2 | 3-4 | 激进 vs 保守 | 策略对比、诱惑风险回报 |
| 3 | 5-6 | 推墙专测 vs 反问专测 | 单功能深度测试 |
| 4 | 7-8 | 极限存活 vs 极限触发突发 | 边界条件、罕见结局 |
| 5 | 9-10 | 完美通关 vs 新手体验 | 最优路径、首次引导 |

### 3. Agent prompt 模板

```
你是一个ClawTrap游戏的测试员。你需要同时扮演**玩家**和**villain**，完整地玩2局游戏。

## 游戏规则
- 15x19迷宫，66步上限，3点HP
- 卡牌系统：relief/temptation/trial/pressure/truth
- 出口条件：2+知识旗帜 且 depth>=6
- 碎片上限5，推墙1碎片(max3)，反问1碎片
- 诱惑：跟随=55%线索/20%碎片/25%陷阱，忽略=安全
- 突发事件：步10后8%几率，坍塌/传送/倒流
- 试炼HP代价：trial1=free, trial2-3=first_fail=1HP, trial4+=every_fail=1HP

## 你的任务

运行引擎生成2局游戏场景：
```bash
cd C:/Users/tatsuya/Documents/ClawTrap
node scripts/ai-play-engine.js {N} > /tmp/game{N}.json
node scripts/ai-play-engine.js {N+1} > /tmp/game{N+1}.json
```

读取JSON，逐回合玩游戏。第{N}局用{策略A}，第{N+1}局用{策略B}。

重点写有决策的回合（试炼、诱惑、压力、突发、推墙、反问），跳过纯移动步。

试炼时：
- 玩家写2-3句真实回答
- Villain从对抗角度判断（严格但公平）
- 失败且有碎片时考虑反问——写反问内容，villain自评

每局输出格式（中文）：
[步N] 位置(x,y) | HP:X 碎片:X | 卡:role/dealer
  [玩家决策] ...
  [Villain判定] ...

局末写总结。最后写总体评价。全部用中文。
```

### 4. 结果汇总维度

每局必须记录：

```
结局: 逃脱/死亡/囚禁
步数/66 | 最终HP | 碎片(获/剩) | 旗帜
试炼: X通过/Y失败 | 通过率
诱惑: X跟随/Y忽略 | 陷阱命中次数
反问: X次(成功Y) | 推墙: X次
突发事件: 坍塌X/传送X/倒流X
死因分析(如死亡): 具体原因链
```

10局汇总统计：
- 逃脱率/死亡率/囚禁率
- 各功能触发率（%的局至少使用过1次）
- 平均碎片获取/消耗
- 试炼通过率（按难度分）
- 死因分布
- 最佳/最差局分析

## ai-play-engine.js 已知问题

引擎是状态生成器，不是完整的游戏模拟器。以下需要agent自行处理：

### 必须由agent自行追踪的状态
- **HP变化**：引擎不扣HP。诱惑陷阱(-1HP)、试炼失败(-1HP per judge.js规则)、影子追击后退(-1HP)、jumpscare超时(-1HP)均由agent判定
- **碎片变化**：引擎标记 `fragment_possible` 但不一定正确生成（有已知bug）。agent应自行掷骰(20%概率)并追踪
- **诱惑结果**：引擎不决定跟随/忽略。agent掷骰决定结果(55%线索/20%碎片/25%陷阱)
- **突发事件**：引擎标记 `sudden_event_possible` 但不生成事件。agent应自行掷骰(8%)并模拟效果

### 试炼HP代价规则（必须严格遵守）
```
trial 1:        失败不扣HP（学习期）
trial 2-3:      首次失败扣1HP，同一试炼重试失败不扣
trial 4+:       每次失败扣1HP
retreat:         免费（但每5次avoidance扣1HP）
god hand:        扣1HP强制通过
counter-question: 扣1碎片，40%基础成功率
```

### 引擎碎片生成bug
`ai-play-engine.js` 第267行 `visited.add()` 在碎片判定(第276行)之前执行，导致移动碎片条件永远为false。

**agent应绕过此bug**：当引擎标记 `fragment_possible=true` 且agent选择了一个未探索方向时，自行掷20%骰子决定碎片是否生成。

|------|-----------|---------|

| 验证迷宫可达性 | ✅ 统计逃脱率 | ✅ 验证体感 |
| 检测死循环/边界case | ✅ 大样本暴露 | ✅ 深度分析单case |

## 常见测试场景模板

### A. 新功能验证
```
策略分配:
- 2局专测该功能（有碎片就用、主动触发条件）
- 2局自由游玩（观察自然使用率）
- 1局完全不用（对照组）
```

### B. 数值调整后验证
```
1. 先跑概率模拟100局，确认数值方向正确
2. 再跑AI对战5局，确认体验质量没下降
```

### C. 试炼题目扩充后验证
```
策略分配:
- 2局激进回答（测试villain判定尺度）
- 2局敷衍回答（测试fail判定是否合理）
- 1局反问密集（测试反问对话质量）
```

### D. 迷宫平衡调整后验证
```
策略分配:
- 2局速通（测试最短路径可行性）
- 2局完全探索（测试66步是否够）
- 1局推墙密集（测试推墙对路径的影响）
```
